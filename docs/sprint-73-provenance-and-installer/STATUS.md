### [T3] FINDING 2026-06-11 20:38 ET — #12 input-accumulation root-caused: xterm@5.5.0 composition/textarea reconstruction, NOT handler stacking; not fixed on main; fix = chokepoint guard + textarea idle-clear + onData dedup

**Symptom fingerprint drives everything.** Brad's 3,042-char payload is a concatenation of cumulative prefixes at WORD granularity (`i think` → `i think there` → `i think there is`…, 21 full copies + 37 `i think` prefixes of a ~110-char message). Each individual emission was therefore small (~5–110 chars); the poison was the SUM of ~37 emissions, each containing the entire buffer-so-far. Two consequences: (1) any mechanism that only duplicates per-keystroke DELTAS cannot produce this shape; (2) a per-chunk size cap alone would never have caught it — the largest single chunk was ~110 chars.

**Exonerated: the reconnect onData stacking (concur with T4-CODEX 20:27 that the defect EXISTS; disagree that it can be the #12 amplifier).** `app.js:2447` (v1.6.0: `app.js:2144`) re-registers `terminal.onData` per reconnect and discards the IDisposable — real leak, real latent landmine, and it gets fixed in this lane. But it cannot have produced Brad's payload: (a) `reconnectSession` is reachable ONLY from the two `ws.onclose` paths (`app.js:589`, `app.js:2440`), so sockets are strictly serialized — at most one OPEN at a time — and every stacked handler closes over its own dead `ws` behind a `readyState === OPEN` guard, sending nothing; (b) even if N handlers double-sent, xterm `onData` events carry only the keystroke delta — N-fold duplication yields `iitthh…`, never cumulative prefixes. Brad's own "arithmetic doesn't match a clean N-handler model" note was correct.

**The actual mechanism (xterm@5.5.0, pinned via CDN at `index.html:392`; evidence from tag-5.5.0 sources).** On composition-style keyboards — every keydown arrives as keyCode 229: Android/iOS soft keyboards, IMEs, dictation, and remote-access keyboard bridges (Brad: remote Chromium via Tailscale, DevTools unavailable — i.e., exactly this input class):
1. **The hidden textarea accumulates the entire message.** xterm clears its helper textarea ONLY on a non-composition Enter/Ctrl+C keydown (`src/browser/Terminal.ts:1066-1068`); keyCode-229 keydowns return early through `CompositionHelper.keydown` (`Terminal.ts:1012-1017`), so mid-message the clear never fires and `textarea.value` grows to the full buffer-so-far.
2. **`_handleAnyTextareaChanges` reconstructs input as `newValue.replace(oldValue, '')`** (`src/browser/input/CompositionHelper.ts:191`), fired per 229-keydown while not composing (`CompositionHelper.ts:109-113`). The moment the keyboard REWRITES text (autocorrect, auto-capitalize, auto-space, predictive word commit — routine at word boundaries), `oldValue` is no longer a substring of `newValue`, `String.replace` matches nothing, and `diff === newValue` — **the ENTIRE accumulated buffer is sent as one data event** (`CompositionHelper.ts:195-196`).
3. **`_finalizeComposition` sends `textarea.value.substring(start[, end])`** with offsets captured at composition boundaries (`CompositionHelper.ts:134,163,168`) — stale/rewritten offsets resend accumulated tails per composition commit (= per word).
Each word boundary that trips (2) or (3) emits the full prefix; the PTY (and Claude Code's input box inside it) accumulates the concatenation → Brad's payload, byte-shape-exact. Root cause lives in xterm, environment-gated by keyboard/IME — which is why it was observed once, organically, and never reproduced on a desktop keyboard.

**Not fixed on main:** the input path is structurally identical v1.6.0 ↔ main (same two `onData` registrations, same guards, same xterm 5.5.0 pin) — no fix commit exists; deliverable is a live fix, not an evidence chain.

**Server side checked, not implicated:** ws `input` accepts from any connected socket by design (`packages/server/src/index.js:3316-3331`, multi-tab); `POST /api/sessions/:id/input` bypasses xterm/onData entirely — confirming Brad's inject workaround is structurally safe. My fix is client-only; the Sprint-63 body-parser hardening is untouched.

**FIX-PROPOSED (landing next, client-only):**
- **F1 — onData dedup (kills Brad's cause-B family + the leak):** register ONE `onData` at panel creation that dereferences `state.sessions.get(id).ws` at send time; delete the `reconnectSession` re-registration (T4's suggested direction — concur).

Update on bug 2 (the input-accumulation half) — root-caused and fixed; ships in the next release.

**What it was.** Your cause (B) — handler accumulation — turned out to exist in our code but to be innocent of *this* bug. The reconnect path did stack one `terminal.onData` handler per WebSocket reconnect (`app.js`, `reconnectSession`) and we've fixed that too, but stacked handlers can only duplicate per-keystroke *deltas* (`iitthh…`), and your payload was cumulative *prefixes* at word granularity. Your note that "the arithmetic doesn't match a clean N-handler model" was exactly right.

The actual mechanism is in xterm.js 5.5.0's composition handling, which fits your environment (remote-accessed Chromium where DevTools were unavailable — i.e., a composition-style keyboard path, where every keydown reaches xterm as keyCode 229):

1. xterm clears its hidden helper `<textarea>` only on a *non-composition* Enter/Ctrl+C keydown (`src/browser/Terminal.ts:1066-1068`). On a composition keyboard that clear never fires mid-message, so the textarea silently accumulates your entire buffer-so-far.
2. For composition keydowns while not composing, xterm reconstructs "what was typed" as `newValue.replace(oldValue, '')` over that textarea (`CompositionHelper.ts:191`). The moment the keyboard *rewrites* text — autocorrect, auto-space, predictive word commit, all routine at word boundaries — `oldValue` is no longer a substring of `newValue`, the replace matches nothing, and the **entire accumulated buffer** is emitted as one input event.
3. A sibling path (`_finalizeComposition`, `CompositionHelper.ts:163-168`) emits `textarea.value.substring(start[, end])` spans with offsets captured at composition boundaries — stale offsets re-emit accumulated tails per word commit.

One full-buffer emission per word boundary, concatenated in the PTY, is exactly your 3,042-char payload. It also explains why it was non-deterministic and unreproducible on a desktop keyboard: it's keyboard/IME-gated, not panel-age-gated. And it explains why your `POST /api/sessions/:id/input` workaround is clean — that path never touches xterm's textarea.

**What shipped.** We can't patch xterm internals (CDN, zero-build client), but every byte of browser input reaches the PTY through one chokepoint we own — the `onData` handler. Three layers:

- **Runaway detector at the chokepoint** (`input-guard.js`, new): your (C)+(D), but shape-aware. The key observation is that a per-chunk size cap would have missed your case — each emission was only ~5–110 chars; the 3,042 chars accumulated *across* ~37 emissions. So the primary detector keys on the structure itself: consecutive multi-char chunks where each is a strict prefix-extension of the previous. Three consecutive links = runaway → suppressed, loud console forensics, and a panel toast telling you to check the input line before pressing Enter. Pastes (DOM `paste` event or bracketed-paste markers) and ESC-prefixed protocol traffic are exempt; an oversize single chunk is held behind a "send anyway" button instead of a blocking confirm.
- **Bounded substrate:** after a short typing lull (250 ms, never mid-composition), we clear xterm's hidden textarea ourselves — the same clear xterm does on Enter, just not gated on a non-composition keydown. The replace/substring fallbacks above can then never reconstruct more than one typing burst.
- **The handler-stacking fix:** one `onData` registration per panel for its lifetime; reconnects swap the socket reference instead of re-registering.

Regression coverage replays your exact payload shape (cumulative word-prefixes of your message) and pins the false-positive cases: the same message typed as 1-char deltas, IME word commits, mouse-wheel bursts, and pastes all pass untouched.

**On your "possibly related" `lastActivity` drift:** real, but a separate mechanism — `lastActivity` is only bumped by PTY *output* (`session.js`), never by input, so a panel receiving input while the agent is quiet goes stale by design. Tracked separately.
