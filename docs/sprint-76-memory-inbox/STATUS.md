### [T3] FINDING 2026-06-12 13:10 ET — promotion-pass design (claim strategy, gate order + numbers, wiring) — posted before implementation

**Shape:** new `src/promote.ts` exporting `promoteInbox(pool, options?, deps?)` from the package root; deployed as a **sibling edge function `supabase/functions/inbox-promote`** (NOT a step inside rumen-tick). Justification: (a) budget isolation — the tick already spends its wall-clock on extract→relate→synthesize→surface with LLM calls; a 25-row inbox batch adds up to 25 embed + 25 Haiku calls of its own; (b) independent cadence — inbox drain can run more/less often than insight generation without coupling; (c) failure isolation — a poisoned-inbox failure class can never starve insight flow; (d) precedent — graph-inference is already a sibling function, so deploy/sync tooling handles N functions. Schedule artifact: `migrations/003_pg_cron_inbox_promote.sql` cloned from 002 (*/15 cadence, same vault-secret pattern, `<project-ref>` placeholder). Deployable ≠ deployed — ORCH applies/deploys at close.

**Claim (idempotent + concurrency-safe):** single-statement `UPDATE memory_inbox SET metadata = metadata || {rumen claim stamp} WHERE id IN (SELECT id FROM memory_inbox WHERE status='pending' AND (never claimed OR claim older than lease) ORDER BY created_at ASC LIMIT $batch FOR UPDATE SKIP LOCKED) RETURNING *`. Statement-level atomicity makes the claim its own transaction; SKIP LOCKED guards simultaneous claims; a 10-min lease (`metadata.rumen.last_claimed_at`) guards overlapping-but-not-simultaneous passes. Correctness does NOT rest on the lease: every terminal write is a CAS (`WHERE id=$1 AND status='pending'`), and the promote CAS rides **the same short transaction as the memory_items INSERT** (withClient BEGIN→INSERT RETURNING id→UPDATE inbox→COMMIT; CAS returns 0 rows → ROLLBACK, the insert vanishes). Exactly-once promotion even under lease-window races; a crash never yields a promoted memory with a pending inbox row nor the reverse. All expensive calls (embed, Haiku) run OUTSIDE any transaction — pgbouncer transaction-mode friendly, no idle-in-tx exposure.

**Precondition (before claiming anything):** BOTH `OPENAI_API_KEY` and `ANTHROPIC_API_KEY` must be present, else the pass logs and returns without claiming a single row. Config absence is not a row failure — without this, a missing key would burn `attempts` across the whole inbox and mass-reject it as attempts-exhausted within 5 passes.
