## Why this sprint (in one paragraph)

Brad upgraded to the latest published packages (termdeck@1.0.0, termdeck-stack@0.6.0, mnestra@0.4.0, rumen@0.4.4) but his database stayed frozen at whenever his project was first kickstarted. Sprint 38 onward: graph-inference Edge Function never deployed, vault key never created, TermDeck migration 003 never applied, Mnestra migrations 009–015 never applied. `rumen-tick` succeeded ~6 days running with `sessions_processed=0, insights_generated=0` every cycle — looked healthy, wasn't. Brad's manual recovery script worked end-to-end (first cron-tagged edge written, similarity 0.8686). The fix is upgrade-aware migration detection in the installer + a symptom-side warning in `mnestra doctor` so the silent no-op pattern doesn't hide schema gaps for days. Both changes ride together as v1.0.1; Sprint 51 cost-panel work resumes as Sprint 52.

## Acceptance criteria

1. **T1 audit-upgrade:** Re-run `termdeck init --mnestra && termdeck init --rumen` against a Sprint-37-era project (or simulate one by dropping migrations 009–015 + the graph-inference function on a test branch). Expected: audit detects all 6 missing artifacts, applies them in order, exits clean. Idempotent — second run reports "nothing to do." **Templating coverage:** audit-upgrade applies Rumen 002 with `<project-ref>` substituted (regression test guards a future bypass).
2. **T1 Edge Function env fallback:** Bundled `rumen-tick/index.ts` and `graph-inference/index.ts` read `Deno.env.get('DATABASE_URL') ?? Deno.env.get('SUPABASE_DB_URL')`. New install on a project that has only the auto-injected `SUPABASE_DB_URL` (no manual secret set) returns 200 from a manual fire. Fixture test in `tests/audit-upgrade.test.js` mocks both env paths.
3. **T2 mnestra doctor:** `mnestra doctor` against Brad's project (post his manual fix) shows all green. Against a deliberately-broken test project (drop `memory_relationships.weight` + suspend `graph-inference-tick`), shows red on both with specific recommendations citing the relevant migration/cron jobname.
4. **T3 GRAPH_LLM_CLASSIFY prompt + per-secret CLI:** Fresh `termdeck init --rumen` on a clean project prompts for the toggle. On Y, `supabase secrets list` shows both vars set; manually firing graph-inference returns `llm_classifications > 0`. On N, the wizard-output README explains the flip command. **Per-secret refactor:** `setFunctionSecrets` issues N independent `supabase secrets set KEY=VAL` invocations (one per key). Test simulates a v2.90.0-style multi-arg drop and verifies the per-call path still lands every secret.
5. **T3 Vault SQL-Editor URL:** Wizard's install-output README never instructs the user to "click Vault." `git grep -i "click.*vault\|vault.*tab" packages/ docs/` returns zero hits. ## Brad 2026-05-03 takeaways (folded into lanes above)

After Brad's CC finished its 4-project install run (jizzard-brain, Structural, aetheria-phase1, aetheria-payroll), it surfaced 5 takeaways for v1.0.1 hotfix scope. All 5 are now folded:

1. **Edge Function `DATABASE_URL` → `SUPABASE_DB_URL` fallback** — bundled source at `packages/server/src/setup/rumen/functions/{rumen-tick,graph-inference}/index.ts` reads `Deno.env.get('DATABASE_URL')` only. Supabase Edge Runtime auto-injects `SUPABASE_DB_URL`; Brad hand-patched all 4 of his deployed copies to `?? Deno.env.get('SUPABASE_DB_URL')`. → **T1.**
2. **Vault dashboard panel missing in current Supabase UI** — wizard text that says "click Vault" is now broken instructions; pivot to a SQL-Editor URL with `vault.create_secret(...)` pre-filled, or automate via `supabase db query --linked`. → **T3.**
3. **`supabase secrets set` v2.90.0 multi-arg parsing unreliable** — sometimes drops secrets, sometimes materializes stray entries (Brad's email parsed as a secret name). One CLI call per secret. → **T3** (`init-rumen.js:setFunctionSecrets` refactor).
4. **Clipboard `\r\n` shred on multi-line `!` pastes** — Brad's terminal converts newlines on paste; bash chokes with `$'echo\r': command not found`. Wizard should never expect users to paste a multi-line block — write to disk and emit a single-line `bash <oneshot.sh>` invocation. → **T4** new Class J + checklist item #11.
5. **Class-A drift root cause confirmed identical on jizzard-brain** — already T1's primary scope. **Bonus:** Rumen migration 002 still ships with raw `<project-ref>` placeholder; the audit-upgrade path needs templating coverage for both Rumen 002 AND 003 (templating IS wired in the fresh-install schedule path at `init-rumen.js:472-505`, but T1's audit-upgrade path is new code and must call `applyTemplating()` too). → **T1.**

Brad's CC also asked an architectural question (one MCP vs four MCPs vs routing-layer wrapper for Mnestra across his 4 Supabase projects). His read: Shape B (one MCP per project, four entries in `~/.claude.json`) for now, revisit Shape C (routing-layer wrapper that picks the Supabase by `cwd`) as a TermDeck v1.1+ feature ask. **Not in Sprint 51.5 scope** — captured as a backlog candidate (`docs/BACKLOG.md` § Sprint 52+ candidates).

## Risks

- **T1 schema-introspection brittleness:** if Brad has manually applied some migrations and skipped others (the `jizzard-brain` recovery scenario), the introspection diff might race or apply migrations in a non-bundled order. Mitigation: each probe maps to exactly one migration; if `weight` exists but `inferred_at` doesn't, apply 009 again with `IF NOT EXISTS` guards (which 009 already has). Safer than nothing.
- **T2 doctor false positives:** if a user has only just provisioned and the cron has only run twice, "all-zeros for 2/10 runs" is normal. Mitigation: require ≥6 cycles before flagging (90 minutes of runtime) — match the soak window Brad's project hid for ~6 days.
- **T3 prompt UX:** users may not know what "AI-classified graph edges" means. Mitigation: link to a short `docs/GRAPH-EDGES-EXPLAINED.md` from the prompt text (NEW, ~30 lines, written T3-side).
